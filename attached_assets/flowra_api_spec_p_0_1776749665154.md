# Flowra API 명세서 (실개발용 상세 버전)

## 1. 문서 개요

이 문서는 Flowra의 P0 기능 구현을 위한 실개발용 API 명세서이다. 프론트엔드, 백엔드, 모바일 앱 개발자가 동일한 기준으로 사용할 수 있도록 다음 내용을 포함한다.

- 엔드포인트 정의
- 인증 방식
- 권한 규칙
- 요청/응답 필드
- 상태값(enum)
- 페이지네이션 규칙
- 공통 에러 코드
- 화면별 연결 API

본 문서는 P0 기준이며, 회사 기능은 일부 P1 확장 항목으로 포함한다.

---

## 2. 기본 규칙

### 2-1. Base URL

`/api/v1`

예시:

- `/api/v1/auth/login`
- `/api/v1/schedules`

### 2-2. Content-Type

- Request: `application/json`
- Response: `application/json`

파일 업로드 API가 추가되면 `multipart/form-data`를 별도 사용한다.

### 2-3. 인증 방식

- JWT Access Token + Refresh Token
- Access Token은 Authorization 헤더에 포함

예시:

```http
Authorization: Bearer {access_token}
```

### 2-4. 시간 형식

모든 datetime은 ISO 8601 문자열 사용

예시:

```text
2026-04-17T14:00:00+09:00
```

### 2-5. ID 타입

모든 주요 ID는 정수형(`number`) 기준으로 정의한다.

### 2-6. 소프트 삭제 여부

P0에서는 단순 삭제로 처리하되, 실제 운영 고려 시 soft delete 확장 가능하도록 설계한다.

## 4. 공통 에러 코드

| code                    | 의미         |
| ----------------------- | ---------- |
| INVALID\_REQUEST        | 잘못된 요청 형식  |
| UNAUTHORIZED            | 인증 필요      |
| FORBIDDEN               | 권한 없음      |
| NOT\_FOUND              | 대상 없음      |
| DUPLICATE\_RESOURCE     | 중복 데이터     |
| VALIDATION\_ERROR       | 유효성 검사 실패  |
| TOKEN\_EXPIRED          | 토큰 만료      |
| TOKEN\_INVALID          | 유효하지 않은 토큰 |
| AI\_PROCESSING\_FAILED  | AI 처리 실패   |
| INTERNAL\_SERVER\_ERROR | 서버 내부 오류   |

예시:

```json
{
  "success": false,
  "message": "유효하지 않은 이메일 형식입니다.",
  "error": {
    "code": "VALIDATION_ERROR",
    "details": {
      "field": "email"
    }
  }
}
```

## 5. 공통 페이지네이션 / 정렬 / 필터 규칙

### 5-1. 페이지네이션 파라미터

- `page`: 기본값 1
- `size`: 기본값 20

- `sort_by`: 정렬 기준 필드

```http
GET /api/v1/tasks?page=1&size=20&sort_by=due_datetime&sort_order=asc
```

## 6. 인증 / 회원 API

## 6-1. 일반 회원가입

**POST** `/auth/signup`

### 설명

일반 이메일/비밀번호 회원가입

### 인증

불필요

### Request Body

| 필드       | 타입     | 필수 | 설명      |
| -------- | ------ | -- | ------- |
| email    | string | Y  | 사용자 이메일 |
| password | string | Y  | 비밀번호    |
| name     | string | Y  | 사용자 이름  |

### Request Example

```json
{
  "email": "user@example.com",
  "password": "Password123!",
  "name": "홍길동"
}
```

### Validation

- email: 이메일 형식
- password: 8자 이상 권장
- name: 1\~30자

### Response

```json
{
  "success": true,
  "message": "회원가입이 완료되었습니다.",
  "data": {
    "user_id": 1,
    "email": "user@example.com",
    "name": "홍길동",
    "login_type": "local",
    "created_at": "2026-04-17T14:00:00+09:00"
  }
}
```

### 오류

- `DUPLICATE_RESOURCE`: 이미 존재하는 이메일
- `VALIDATION_ERROR`: 입력값 오류

- 공통 응답 포맷을 component schema로 분리할 것
- 인증이 필요한 API는 security scheme 지정할 것

- page/size 파라미터 공통화 가능

## 19. 추가 권장 사항

### 권장 1. 메모 생성 후 AI 파싱 처리 방식

실서비스 구조상 `POST /memos`와 AI 파싱은 분리하는 것이 안정적이다. 다만 UX 편의를 위해 `auto_parse=true` 옵션을 두고 서버에서 비동기 처리하는 방식이 적절하다.

### 권장 2. TODO 완료 API 분리

`PATCH /tasks/{taskId}/complete`는 프론트에서 자주 사용되므로 별도 엔드포인트 유지가 편하다.

### 권장 3. 브리핑은 저장 테이블보다 집계 API 우선

P0 단계에서는 별도 briefing 테이블 없이 동적 생성하는 것이 더 단순하고 적절하다.

### 권장 4. 메모 수정 시 재분석 정책 명확화

메모 본문이 바뀌면 parse\_status를 다시 `pending`으로 변경하는 규칙을 팀 내에서 고정하는 것이 좋다.
